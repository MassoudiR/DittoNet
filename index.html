Getting Started

Usage Guide

This guide walks through the complete workflow for intercepting and modifying live mobile traffic — from installing the app to creating your first rule. No prior experience with proxy tools is required.

What You Need

  • An Android device (physical or emulator) — no root required
  • The Ditto Mobile APK installed on the device (download from GitHub Releases)
  • A computer on the same local network, running Python 3.10+
  • The Ditto Server running on that computer (see Quickstart)

How It Works — Overview

Ditto Mobile is a full browser. You use it to visit websites and web apps just like a normal browser. Behind the scenes, every HTTP request the browser makes passes through DittoNet's interception engine before it reaches the internet. You can define rules that tell the engine what to do with those requests — block them, redirect them, or modify their contents — in real time, without touching the target application.

Step-by-Step Example: Modify an API Response

In this example, a web app returns a JSON response containing {"account_tier": "standard"}. We will create a rule that automatically changes it to {"account_tier": "premium"} on every response, without modifying the app or the server.

1 Start the server and open the dashboard

On your computer, run the Python server:

cd "Ditto Server"
.venv\Scripts\activate
python test_implementation.py

Open http://localhost:5000 in your browser. You will see the DittoNet dashboard with a live traffic log and a Rules panel.

2 Connect Ditto Mobile to the server

Open Ditto Mobile on your Android device. Tap the Settings icon and enter your computer's local IP address (e.g., 192.168.1.10) in the Remote Server field. Tap Connect.

The status indicator turns green when the device successfully reaches the server via GET /api/health. All traffic from the browser will now be routed through the server's rule engine.

3 Create a MATCH_REPLACE rule on the dashboard

In the dashboard, go to Rules → New Rule and fill in the following fields:

URL Pattern*api.example.com/v1/account*
MethodANY
PhaseResponse
ActionMATCH_REPLACE
Match"account_tier": "standard"
Replace"account_tier": "premium"

Click Save. The rule is immediately active — no restart required.

4 Browse normally — the rule activates automatically

In Ditto Mobile, navigate to the target website. When the browser sends a request matching *api.example.com/v1/account*, the server intercepts the response and replaces "account_tier": "standard" with "account_tier": "premium" before the browser renders it.

The dashboard updates in real time, highlighting the modified transaction with a MODIFIED badge. You can click any log entry to inspect the original and mutated headers and body side by side.

Other Common Use Cases

BLOCKRemove ads and trackers

Create a BLOCK rule with a URL pattern matching your ad network domain (e.g., *googlesyndication.com*). Every request to that domain will be dropped and returned as HTTP 403, preventing the resource from loading.

REDIRECTSwitch API environments

Redirect production API calls to a local mock server by creating a REDIRECT rule targeting your production domain. Set replaceStr to your local address (e.g., http://192.168.1.10:8080).

HEADER_INJECTInject authentication tokens

Add an Authorization header to every outbound request without modifying the app. Useful for testing APIs that require bearer tokens during development sessions.

BODY_REPLACESwap entire response payloads

Replace the complete JSON body of an API response with a custom fixture. Useful for testing UI behaviour against edge-case or error-state API responses without a staging server.

STANDALONE MODE   If you do not have a computer available, Ditto Mobile can operate entirely on-device. Go to Settings → Mode → Standalone. Rules are created directly in the app and evaluated locally with no server dependency. The dashboard and Python plugins are not available in this mode, but all six rule action types still work.
Interception Engine

System Architecture

DittoNet intercepts HTTP/HTTPS traffic at the WebResourceRequest boundary inside the Flutter WebView runtime, before requests reach the Android OS network stack. This position in the call chain bypasses SSL/TLS certificate pinning enforced at the application layer, as the interception occurs within the trusted WebView execution context before any TLS validation.

Execution Flow

1
WebResourceRequest Capture
The Flutter InAppWebView widget calls shouldOverrideUrlLoading and shouldInterceptRequest for every outbound navigation and resource fetch. DittoNet's InterceptorCore captures these callbacks before any network I/O is initiated.
2
Mode Determination
The engine checks server reachability. If the Ditto Server is available, the full request payload is serialized and forwarded via POST /api/intercept/request. Otherwise, the local Dart rules engine evaluates the request in-memory with no network dependency.
3
Rule Evaluation & Mutation
Matching rules are applied in deterministic precedence order: BLOCK → REDIRECT → EXECUTE_PLUGIN → BODY_REPLACE → MATCH_REPLACE → HEADER_INJECT. Terminal actions (BLOCK, REDIRECT) cause immediate return. Non-terminal actions accumulate across all matched rules for the same request.
4
Response Phase & Telemetry
After the upstream server responds, the payload is submitted to POST /api/intercept/response for response-phase rule evaluation. The server then emits a Socket.IO event to update the dashboard and appends the complete transaction to the rolling in-memory log.

engine.py — InterceptEngine.process_request()
def process_request(self, payload):
    url, method = payload.get("url",""), payload.get("method","")
    headers = dict(payload.get("headers", {}) or {})
    body    = payload.get("body", "")
    flow_id = payload.get("flowId")

    # Phase 1: Programmatic inspectors (run before SQLite rules)
    for insp in self.inspectors:
        if insp['phase'] not in ('Request', 'Both'): continue
        if self._match_pattern(insp['pattern'], url):
            result = insp['func'](flow_id, "Request", headers, body)
            if result and isinstance(result, tuple) and len(result) == 2:
                headers, body = result

    # Phase 2: SQLite rules in deterministic precedence order
    for rule in self._get_matching_rules(url, method, "Request"):
        action = rule.get("actionType")
        if   action == "BLOCK":
            return {"action":"BLOCK","modified":True,"headers":headers,"body":body}
        elif action == "REDIRECT":
            return {"action":"REDIRECT","redirectUrl":rule.get("replaceStr",""),
                    "modified":True,"headers":headers,"body":body}
        elif action == "MATCH_REPLACE":
            body = self._apply_match_replace(body,rule.get("matchStr"),rule.get("replaceStr"))
        elif action == "BODY_REPLACE":
            body = json.loads(rule.get("replaceStr","{}"))
        elif action == "HEADER_INJECT":
            k,v = rule.get("matchStr","").split(":",1)
            headers[k.strip()] = v.strip()
        elif action == "EXECUTE_PLUGIN":
            _,headers,body = self._execute_plugin(rule.get("pluginName"),flow_id,"Request",headers,body)

    return {"action":"ALLOW","modified":False,"headers":headers,"body":body}
interceptor_core.dart — local rule evaluation
Future<WebResourceResponse?> _evaluateLocalRules(WebResourceRequest request) async {
  final url = request.url.toString();
  for (final rule in state.activeRules.where((r) => r.isEnabled)) {
    if (!rule.matches(url)) continue;
    switch (rule.actionType) {
      case ActionType.BLOCK:
        return WebResourceResponse(
          statusCode: 403,
          reasonPhrase: 'Blocked by DittoNet Rule #${rule.id}',
          headers: {'Content-Type': 'text/plain'},
          data: Uint8List.fromList(utf8.encode('Request blocked.')),
        );
      case ActionType.REDIRECT:
        await _controller.loadUrl(urlRequest: URLRequest(url: Uri.parse(rule.redirectUrl!)));
        return WebResourceResponse(statusCode: 302, headers: {});
      default:
        break;
    }
  }
  return null; // No terminal rule matched — allow unmodified
}
Interception Engine

Rule Action Reference

Rules are stored in SQLite and matched against traffic using wildcard patterns (* expands to .* in regex). Each rule has a phase (Request | Response | Both) and a method filter (ANY or a specific HTTP verb).

BLOCKTerminal · Request & Response

Immediately terminates request dispatch and returns a synthetic HTTP 403 to the WebView. No upstream connection is made. Subsequent rules for the same request are not evaluated.

return {"action": "BLOCK", "modified": True, "headers": headers, "body": body}
REDIRECTTerminal · Request phase

Returns a redirect instruction. The redirectUrl field in the response contains the value of the rule's replaceStr. The mobile client loads the alternate URL in the WebView.

return {"action": "REDIRECT", "redirectUrl": rule.get("replaceStr"), "modified": True}
MATCH_REPLACENon-terminal · Request & Response

Performs exact-string substitution on the body. CRLF sequences are normalized to \n before matching. If the body is a JSON object (dict), it is serialized to a string before substitution and re-parsed afterward.

def _apply_match_replace(self, body, match_str, replace_str):
    if isinstance(body, str):
        return body.replace('\r\n','\n').replace(match_str, replace_str)
    body_str = json.dumps(body).replace('\r\n','\n').replace(match_str, replace_str)
    return json.loads(body_str)
BODY_REPLACENon-terminal · Request & Response

Replaces the entire body with the content of replaceStr. When the original body was JSON, the replacement is parsed first with json.loads(), falling back to ast.literal_eval() for Python dict literals.

try:
    body = json.loads(rule.get("replaceStr", ""))
except Exception:
    body = ast.literal_eval(rule.get("replaceStr", ""))  # dict-literal fallback
HEADER_INJECTNon-terminal · Request & Response

Appends or overwrites a single HTTP header. The matchStr field must use the format Header-Name: value. Key and value are split on the first colon and whitespace-stripped.

k, v = rule.get("matchStr", "").split(":", 1)
headers[k.strip()] = v.strip()
# Example: "Authorization: Bearer eyJhbGciOiJIUzI1NiJ9..."
EXECUTE_PLUGINNon-terminal · Request & Response

Invokes a named Python function registered via @server.plugin(name). The plugin receives (flow_id, phase, headers, body) and must return (headers, body). The rule's pluginName must exactly match the registered name. Unregistered plugin names log a warning and return the payload unmodified.

@server.plugin("NormalizeCoins")
def normalize_coins(flow_id, phase, headers, body):
    if phase == "Response" and isinstance(body, dict) and "coins" in body:
        body["coins"] = 999999
    return headers, body

JavaScript Hooks & Workspace Sync

JavaScript hooks are injected at the DOCUMENT_START lifecycle phase — before the target page's scripts execute or the DOM tree is constructed. Hooks and local rules are bundled in a workspace document, exchanged between mobile and server via GET /api/sync/workspace.

Workspace Schema

{
  "timestamp": 1751384580000,
  "local_rules": [
    {
      "id": "rule-001",
      "targetPattern": "*api.target.com/v1/*",
      "method": "ANY",
      "phase": "Response",
      "actionType": "MATCH_REPLACE",
      "matchStr": "\"role\":\"user\"",
      "replaceStr": "\"role\":\"admin\"",
      "isEnabled": true
    }
  ],
  "js_hooks": [
    {
      "id": "hook-001",
      "name": "Disable WebDriver Flag",
      "targetOrigin": "*",
      "isEnabled": true,
      "script": "Object.defineProperty(navigator,'webdriver',{get:()=>false});"
    },
    {
      "id": "hook-002",
      "name": "Override Geolocation",
      "targetOrigin": "https://maps.target.com",
      "isEnabled": true,
      "script": "navigator.geolocation.getCurrentPosition=(cb)=>cb({coords:{latitude:36.8065,longitude:10.1815}});"
    }
  ]
}
Hooks are applied via evaluateJavascript immediately after the onPageStarted event fires, guaranteeing execution before DOMContentLoaded and before the page's own <script> tags run. A targetOrigin value of "*" applies the hook to all origins.
Extensibility

Python Plugin API

Inspectors execute automatically on every matching request or response and run before SQLite rules. Plugins are registered by name and invoked only when a dashboard rule with EXECUTE_PLUGIN references them.

Inspector Decorator

@server.inspector(pattern, method="ANY", phase="Both")
@server.inspector(pattern="*api/v1/auth*", method="POST", phase="Both")
def audit_auth_traffic(flow_id, phase, headers, body):
    """Logs every auth request/response and tags responses."""
    print(f"[{phase}] Auth flow: {flow_id}")
    if phase == "Response" and isinstance(headers, dict):
        headers["X-Ditto-Audited"] = "true"
    return headers, body  # Must always return (headers, body)

Plugin Decorator

@server.plugin(name)
@server.plugin("OverrideAccountTier")
def override_account_tier(flow_id, phase, headers, body):
    """
    Assign to a rule via the dashboard (actionType: EXECUTE_PLUGIN,
    pluginName: OverrideAccountTier) targeting account/session endpoints.
    """
    if phase == "Response" and isinstance(body, dict):
        if "account_tier" in body:
            body["account_tier"] = "enterprise"
    return headers, body  # Must always return (headers, body)

if __name__ == "__main__":
    server.run()  # Binds to 0.0.0.0:5000
NOTE  If a rule references a plugin name that is not registered, the engine logs a warning and returns the payload unmodified without raising an exception.

Quickstart

Deploy the Python server and build the Android client in two steps. No root access, system certificates, or iptables configuration required.

1 Deploy Ditto Server

Creates a Python virtual environment, installs Flask and Flask-SocketIO, and launches the server. Dashboard available at http://localhost:5000.

cd "Ditto Server"
python -m venv .venv
.venv\Scripts\activate        # Windows
# source .venv/bin/activate   # macOS / Linux
pip install -r requirements.txt
python test_implementation.py
2 Build Ditto Mobile

Fetches Dart/Flutter dependencies, runs static analysis, and produces a release APK. Deployable to any Android device without root or a custom recovery.

cd "Ditto Mobile"
flutter pub get
flutter analyze --no-fatal-infos
flutter build apk --release
# Output: build/app/outputs/flutter-apk/app-release.apk
3 Connect & Configure

In Ditto Mobile, enter the server's local IP in Remote Bridge mode. The app verifies connectivity via GET /api/health and syncs the workspace via GET /api/sync/workspace.
